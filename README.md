# 社長ランチごちします - AI写真判定 + freee連携 福利厚生費管理SaaS

**GAIS AI DEVCON 2026 応募作品**

---

## 概要

**社長ランチごちします**は、現場で働く従業員（職人・ドライバーなど）の食事補助（福利厚生費）を、**写真1枚で申請・管理**できるSaaS型システムです。

### コアバリュー

- **写真を撮るだけ**: 弁当の写真を撮影して申請。ログイン不要
- **AIが不正を自動検知**: 食事と無関係な写真はAIが自動で「要確認」フラグを付与。経理向けに判定理由を日本語で生成
- **freee連携**: 月末にまとめてfreee会計に下書きを自動作成
- **業務に自然に浸透**: 職人の操作は変わらない。AIは裏側で動き、経理の確認工数だけが減る

---

## 生成AIの活用（審査ポイント）

本作品では、**OpenAI Vision API** を使った**写真妥当性チェック + 判定理由の自然言語生成**を実装しています。

### なぜ「レシートOCR」ではなく「写真妥当性チェック」なのか

現場の実態を考えると：
- 職人が撮るのは**弁当そのものの写真**であり、レシートではない
- レシートを撮影させるなら手入力と手間が変わらない
- 本当に必要なのは「**その写真が食事の証跡として妥当か**」の判定

### AI機能の詳細

| 機能 | 説明 | AIの必然性 |
|------|------|-----------|
| **食事写真の判定** | 弁当/外食/コンビニ/飲み物のみ/食事以外を自動分類 | ルールベースでは画像内容の判定は不可能 |
| **判定理由の生成** | 経理向けに「なぜOK/NGか」を日本語で説明 | **生成AIの「生成」能力が不可欠**。テンプレートでは対応不可能な多様な写真に対応 |
| **注意フラグの付与** | 「飲み物のみ」「不鮮明」「高級店の可能性」等を自動付与 | 経理の全件チェックを不要にする |
| **自動レビュー振り分け** | AIがNGまたはフラグ付きの申請を自動で「要確認」に | 経理はフラグ付きだけチェックすればよい |

### AI活用のフロー

```
[職人] いつも通り弁当の写真を撮って申請
    ↓
[AI] 写真を判定 → 「弁当の写真です。問題ありません」
    ↓  or
[AI] 写真を判定 → 「建設現場の写真です。食事が映っていません」→ 要確認フラグ
    ↓
[管理画面] 経理はAIコメントを見て判断（要確認だけチェック）
    ↓
[月次集計] freee下書き作成
```

### なぜ生成AIが必然なのか

1. **画像認識はルールベースでは不可能**: 「弁当の写真かどうか」を判定するには、画像の内容を理解するAIが必要です。色味やピクセルパターンでは判定できません。
2. **判定理由の生成はテンプレートでは対応不可能**: 撮影される写真は千差万別です。「コンビニ弁当の写真です」「定食屋の食事写真です」「工具の写真のようです」など、**内容に応じた説明を自然言語で生成**できるのは生成AIだからこそです。
3. **職人の操作は変わらない**: AIは裏側で動くため、現場の人に新しい操作を覚えてもらう必要がありません。業務に**自然に浸透**します。

## セットアップ手順

### 1. 環境変数を設定

`docs/env.example.txt` を参考に `.env.local` を作成してください。

```bash
cp docs/env.example.txt .env.local
# 各値を編集
```

#### 必須
- `MONGODB_URI` - MongoDB Atlas接続文字列
- `NEXTAUTH_SECRET` - セッション暗号化キー
- `ADMIN_EMAIL` / `ADMIN_PASSWORD` - 管理者ログイン

#### AI機能（推奨）
- `OPENAI_API_KEY` - OpenAI APIキー
- `OPENAI_BASE_URL` - APIエンドポイント（任意）
- `OPENAI_MODEL` - 使用モデル（任意、デフォルト: gpt-4o）

#### freee連携（任意）
- `FREEE_ACCESS_TOKEN` / `FREEE_COMPANY_ID` など

#### 写真ストレージ（任意）
- `GOOGLE_OAUTH_*` - Google Drive OAuth設定

### 2. 起動

```bash
npm install
npm run dev
```

### 3. 動作確認

1. `http://localhost:3000/api/auth/signin` で管理ログイン
2. `http://localhost:3000/admin/pairing` でQR発行
3. 職人端末でQRを読み取りペアリング
4. `http://localhost:3000/worker/entry` で写真付き申請
   - 弁当の写真を撮影 → AIが自動判定 → 結果表示 → 申請
5. `http://localhost:3000/admin/entries` で申請一覧（AI判定結果付き）
6. `http://localhost:3000/admin/monthly` で月次集計 → freee下書き作成

## 画面構成

| 画面 | パス | 機能 |
|------|------|------|

| 画面 | パス | 機能 |
|------|------|------|
| ダッシュボード | `/admin` | 今月の概要 |
| QR発行 | `/admin/pairing` | 従業員追加・ペアリングQR発行 |
| 申請一覧 | `/admin/entries` | AI判定列・要確認フィルタ |
| 申請詳細 | `/admin/entries/[id]` | AI判定理由・フラグ・承認操作 |
| 月次集計 | `/admin/monthly` | プレビュー → freee下書き作成 |

## AI判定の詳細仕様

### 入力
- JPEG/PNG等の画像ファイル（最大10MB）

### 出力（JSONスキーマ）

```json
{
  "is_valid_meal": true,
  "detected_category": "bento",
  "reason": "コンビニ弁当の写真です。食事補助の申請として問題ありません。",
  "description": "プラスチック容器に入った幕の内弁当が映っています。",
  "flags": []
}
```

### カテゴリ
| カテゴリ | 説明 | AI判定 |
|---------|------|--------|
| `bento` | 弁当・おにぎり・サンドイッチ | OK |
| `restaurant` | レストラン・食堂での食事 | OK |
| `convenience_store` | コンビニの食品 | OK |
| `drink_only` | 飲み物のみ | NG（要確認） |
| `receipt` | レシートのみ（食事が映っていない） | NG（要確認） |
| `unrelated` | 食事と無関係 | NG（要確認） |
| `unclear` | 不鮮明・判定困難 | NG（要確認） |

### グレースフルデグレード
- `OPENAI_API_KEY` 未設定時: AI判定をスキップし、従来の手動確認モードで動作
- AI判定失敗時: エラーを通知しつつ、申請は継続可能
