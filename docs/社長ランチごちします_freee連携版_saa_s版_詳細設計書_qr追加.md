# 社長ランチごちします freee連携版（SaaS版）詳細設計書

※本書はハッカソン提出用の **SaaS版 詳細設計書（QR対応版）** である。

---

## 1. 目的

本アプリは、現場で働く職人が利用する **仕出し弁当補助** を、

- 現場負担を最小化し（ログイン不要）
- 経理・税務上の判断を管理側に集約し
- freee会計と安全に連携する

ことを目的とする。

---

## 2. 前提条件（ハッカソン版）

- freee連携設定は **.env.local に固定**（画面設定なし）
- 写真ストレージは **Google Drive に固定**（環境変数＋コード内設定、画面設定なし）
- 職人UIは **完全ログインレス**
- 管理UIのみ NextAuth によるログイン

---

## 3. 全体アーキテクチャ

- フロントエンド / BFF：Next.js 15（App Router）
- 認証：NextAuth.js（管理UIのみ）
- DB：MongoDB Atlas
- ストレージ：Google Drive（証跡用）
- デプロイ：Vercel

---

## 4. ユーザー種別

| 種別 | 説明 |
|---|---|
| 職人 | ログイン不要。QRで端末登録後、写真申請のみ行う |
| 管理者 | 経理・管理担当。承認・集計・freee連携を行う |

---

## 5. アプリ専用QR機能（端末ペアリング）【追加】

### 5.1 目的

- 職人に ID・パスワード入力をさせない
- 端末とユーザーを **安全に1対1で紐づける**
- なりすまし・使い回しを防止

QRコードは「端末登録のための一時鍵」として利用する。

---

### 5.2 QRコード仕様

QRコードには以下の **token付きURL** を埋め込む。

```
https://{app-domain}/pairing?token={UUID}
```

#### tokenの性質
- UUID形式
- 1回限り有効
- 有効期限付き（例：24時間）
- 使用後は即失効

#### 再発行ルール（重要）
- 管理者が「再発行」を行った場合、**旧tokenは即失効**（以降は使用不可）
- 旧tokenを読み取った端末はペアリングできない（「無効なQRコードです」扱い）

---

### 5.3 関連テーブル設計

#### device_pairing_tokens

| カラム | 説明 |
|---|---|
| token | QRに埋め込む一意トークン |
| tenant_id | テナントID（SaaS分離のため必須） |
| user_id | 紐づけ対象ユーザー |
| expires_at | 有効期限 |
| used_at | 使用日時（null=未使用） |
| created_by | 管理者ID |
| created_at | 作成日時 |

#### devices

| カラム | 説明 |
|---|---|
| device_id | 端末ID |
| tenant_id | テナントID（SaaS分離のため必須） |
| device_secret | 端末に払い出す秘密値（HTTP-only Cookieで保持） |
| device_fingerprint | 参考情報（任意）。監査・異常検知用で、認証の主キーにはしない |
| user_id | 紐づくユーザー |
| paired_at | ペアリング日時 |

---

### 5.4 QR発行フロー（管理UI）

```
[管理ダッシュボード]
   ↓
[QR発行]
   ↓
[QR表示]
```

1. 管理者が対象ユーザーを選択
2. サーバー側で token を生成し DB 保存
3. QRコードを画面表示
4. 職人がスマホで読み取り

※ ハッカソン版では「画面表示のみ」で運用

---

### 5.5 ペアリングフロー（職人UI）

```
[QR読み取り]
   ↓
[トークン検証]
   ↓
[端末登録]
   ↓
[ペアリング完了]
```

処理内容：
- token 有効性チェック
- device_secret を発行し、HTTP-only Cookie として端末に保存
- （任意）device_fingerprint を生成して参考情報として保存
- devices テーブルに登録
- token.used_at を更新

---

### 5.6 エラー処理

| ケース | 表示メッセージ |
|---|---|
| token不正 | 無効なQRコードです |
| 期限切れ | QRコードの有効期限が切れています |
| 使用済み | このQRは既に使用されています |

## 6. 申請フロー（職人UI）

- ペアリング済み端末のみ申請可能
- 弁当またはレシート写真を撮影
- lunch_entries に明細ログとして保存

---

## 7. 管理フロー（管理UI）

- 申請一覧確認
- 要確認・重複チェック
- 月次集計（手動）
- freeeへ下書き登録

---

## 8. freee連携（ハッカソン版）

- .env.local にて以下を保持
  - FREEE_CLIENT_ID
  - FREEE_CLIENT_SECRET
  - FREEE_COMPANY_ID
- freeeへ **下書き作成**（勘定科目ID・税区分の初期値は **MVPでは未固定**）
- 技術検証で「福利厚生費として登録」できる手順が確定したら、環境変数等で指定可能にする
- 最終確定は freee 側で実施

---

## 9. セキュリティ・監査観点

- 端末ペアリングは1端末1ユーザー
- QR token の再利用不可
- lunch_entries は履歴ログとして不変
  - 写真は **元画像（オリジナル）を保持**（監査での再確認用）
  - アクセス権は最小（公開リンクは使わない前提）
  - photo_hash（SHA-256）を保存し、改ざん検知に用いる
