# 社長ランチごちします（freee連携 / SaaS）設計書（作り直し版）

ハッカソンの〆切延長（約1か月）に合わせて、現状の「提出用の最小実装」を**プロダクトとして運用できる品質**へ作り直すための設計方針をまとめる。

---

## 0. この設計書のゴール

- 現場（職人）を止めずに「**写真を撮って送るだけ**」で入力できる
- 経理は月末に「**確認してfreeeで承認するだけ**」に近づける
- 監査/税務で説明できるよう「**証跡（写真/入力/操作ログ）**」を確実に残す
- SaaSとして複数社（テナント）を安全に分離し、**設定・権限・連携先**をテナント単位で管理する

---

## 1. 現状（ハッカソン版）の前提と限界

既存ドキュメント・実装は「ハッカソン最小」の前提が強い。

### 1.1 ハッカソン前提（現状の仕様）
- freee/Google Driveの設定が **`.env.local`（Vercel env）固定**で、テナント別に設定できない
- 管理ログインは **Credentials（`ADMIN_EMAIL`/`ADMIN_PASSWORD`）** の簡易方式
- 職人は **ログインレス**（`device_secret` Cookie）で識別
- 月次は **管理画面から手動トリガー**（ジョブ化/自動化なし）
- 申請は **写真が任意**、Driveアップロード失敗時も申請を受け付ける（緊急モード）

### 1.2 このままでは困る点（作り直し対象）
- **テナント分離**が実質 `DEFAULT_TENANT_ID` 依存で、増やせない
- **連携（freee/Drive）の運用**がテナントごとの設定・再認可・失効対応を想定していない
- **写真が任意**だと証跡が弱く、監査/税務/不正対策の芯が抜ける
- 月次処理が手動前提で、失敗時の再実行・冪等・運用手順が不足
- 権限（RBAC）が最小で、経理/閲覧者/管理者などの役割分離が弱い

## 2. ToBe要件（作り直し後の完成形）

### 2.1 機能要件（Must）

#### A. テナント（会社）管理
- テナント作成（オンボーディング：**管理者（運営）が作成**）
- テナント設定（補助単価/月上限/締め日/通貨/タイムゾーンなど）
- テナントごとの社員（職人）管理（追加/無効化/名寄せ）

#### B. 職人の端末ペアリング（QR）
- 管理者が社員に対して **ペアリングQR** を発行
- QRは **有効期限 + 1回限り + 再発行で旧トークン失効**
- 端末側は `device_secret`（HTTP-only cookie）を受け取り、以後ログインレス
- 端末紛失/退職に備え、**端末の無効化/再ペアリング** ができる

#### C. 日次入力（職人UI）
- 入力項目（基本）  
  - 利用日、現場名、金額、写真、備考（任意）
- **写真は原則必須**（「撮って送るだけ」が価値の中心）
- 画像の拡張子/サイズ制限、EXIF扱い（方針を決める：保持 or 削除）
- 重複検知（SHA-256等のハッシュ）で `needs_review` を付与（処理は止めないが、**写真は残る**）

#### D. 管理UI（経理/管理者）
- 申請一覧（フィルタ：月、社員、現場、ステータス）
- 申請詳細（写真、入力、重複判定、操作ログ）
- レビュー（承認/要確認/却下/差戻し ※要件に応じて）
- 月次集計（プレビュー→実行、冪等キー、再実行、失敗理由の表示）
- CSV/Excelエクスポート（監査/提出用）

#### E. freee連携（テナント別）
- テナント管理者がUIから **freee接続（OAuth）** を完了できる
- 会社選択（`company_id`）、勘定科目/税区分/補助科目などをUIで設定できる
- 月次集計結果から freeeに **取引（支出）の下書き** を作成
- 失敗時は理由を保持し、**再実行可能**（冪等/二重計上防止）

#### F. 写真ストレージ（テナント別）
「Google Drive継続」か「S3等へ移行」かを明示して設計する（後述）。
- 写真は原本を保持し、申請と1対1に紐づく
- 管理UIから安全に閲覧できる（公開リンクを安易に使わない）

### 2.2 非機能要件（Must）
- **テナント境界の強制**（すべての読み書きに tenant_id を付与・検証）
- **権限（RBAC）**：admin/accountant/viewer 等、画面/APIの権限を統一的に制御
- **監査ログ**：重要操作（QR発行、ペアリング、申請作成、レビュー変更、月次実行、freee送信）を必ず記録
- **冪等性**：月次実行やfreee送信は二重作成を防ぐ
- **運用性**：エラーの可視化、再試行、バックアップ、鍵のローテーション

## 3. 全体アーキテクチャ（ToBe）

### 3.1 採用スタック（現行をベースに強化）
- Next.js（App Router） + Server Actions/Route Handlers
- MongoDB Atlas（マルチテナント）
- ストレージ（要決定）：Google Drive もしくは S3互換（Cloudflare R2等）
- freee API（OAuth 2.0 Code Flow）
- デプロイ：Vercel
- ジョブ：Vercel Cron + 外部キュー（例：Upstash/QStash）または内部の再実行可能キュー（DB駆動）

### 3.2 レイヤ分割（責務）
- **UI層**：職人UI/管理UI
- **API/Action層**：入力・操作の受け口（認可・バリデーション・リクエストID付与）
- **ドメイン層**：集計/重複判定/レビュー状態遷移/freee送信などのビジネスロジック
- **インフラ層**：DB/ストレージ/freee/Drive へのアダプタ

## 4. データ設計（ToBe：追加/変更）

### 4.1 追加コレクション（新規）

#### `tenants`
- `tenant_id`（PK）
- `name`
- `status`（active/inactive）
- `timezone`（例：Asia/Tokyo）
- `created_at`, `updated_at`

#### `tenant_settings`
- `tenant_id`（PK）
- `subsidy_unit_yen`
- `monthly_limit_yen`
- `closing_day`（月締めルール）
- `require_photo`（原則 true）
- `created_at`, `updated_at`

#### `admin_users`
- `tenant_id`
- `email`
- `role`（admin/accountant/viewer）
- `status`
- `created_at`, `updated_at`

#### `integrations_freee`
- `tenant_id`（PK）
- `connected`（boolean）
- `company_id`（number）
- `account_item_id`（string/number）
- `tax_code`（number）
- **token保管（後述：暗号化）**
- `created_at`, `updated_at`, `last_error`

#### `integrations_storage`
- `tenant_id`（PK）
- `provider`（drive / s3 / r2 ...）
- `config`（暗号化/秘匿領域を分離）
- `created_at`, `updated_at`, `last_error`

### 4.2 既存コレクションの見直し（例）

#### `lunch_entries`
- `photo_*` を「緊急モードのoptional」から、ToBeでは原則必須へ（例外時は `review_status` で扱う）
- `status`（submitted / approved / rejected / needs_review など）を明確化
- `source`（worker_web / admin_import など）を追加

#### `freee_exports`
- `export_batch_id`（ユニーク）で冪等制御
- `status`（previewed/executed/succeeded/failed）を細分化
- `error_message` と `error_meta`（APIレスポンス要旨）を保持

### 4.3 インデックス（最低限）
- 全テーブル：`tenant_id` を先頭にした複合インデックス
- `lunch_entries`: `(tenant_id, year_month, employee_id, entry_date)`
- 重複検知：`(tenant_id, photo_hash)`（存在する場合のみ）
- `freee_exports`: `(tenant_id, export_batch_id)` をユニーク

## 5. 認証・認可（ToBe）

### 5.1 管理側（Admin）
- NextAuthは継続しつつ、認証方式を段階的に強化
  - Phase1：現行Credentialsを維持しつつ `admin_users` + role をDBで管理
  - Phase2：Google OAuth / Magic Link 等を導入（組織の運用に合わせる）

### 5.2 職人側（Worker）
- 端末ペアリング（QR）で `device_secret` を払い出し
- `device_secret` は
  - HTTP-only / Secure / SameSite の方針を明確化
  - DB側には **ハッシュ化**して保存（現状の方向性を維持）
- 端末の無効化・再発行・監査を可能にする

### 5.3 テナント境界
- すべての読み書きに `tenant_id` を強制
- 管理セッション/Workerセッションから `tenant_id` を必ず導出し、入力からは受け取らない

## 7. freee連携の再設計（ToBe）

### 7.1 OAuth方針
- **OOB（`urn:ietf:wg:oauth:2.0:oob`）依存を脱却**し、プロダクションはHTTPSコールバックに統一する
  - 例：`https://{domain}/api/integrations/freee/callback`
- 接続フロー
  1. 管理UIで「freeeに接続」
  2. freeeの認可画面 → コールバックで `code` を受領
  3. サーバでトークン交換（access/refresh）
  4. **refresh tokenのローテーション**をDBに保存

### 7.2 トークン保管（重要）
- `refresh_token` はDBに平文で置かない（最低限どちらかを採用）
  - A案：Vercel KMS相当（外部KMS）で暗号化してDBに保存
  - B案：アプリ内の暗号鍵（`TOKEN_ENCRYPTION_KEY`）で暗号化して保存（鍵管理を別途設計）
- 監査ログにはトークンを絶対に出さない（既存loggerのredactionを前提にキー名も統一）

### 7.3 月次→freee下書き作成の仕様
- 作成単位（確定）：**テナント×月で1件**
  - 経理が確認する下書き件数を抑え、承認/修正フローを簡単にする
  - 社員別の内訳は下書きの摘要（description）や添付（リンク）に集約する
- 下書き本文（description）に必ず残す
  - 対象月、件数、補助合計、参照リンク（管理画面URL/写真への導線）
- 冪等性
  - `export_batch_id = tenant_id + year_month` をユニーク化
  - freee送信に失敗した場合は `status=failed` で保持し、同一batchの再実行でリトライ

## 8. 写真ストレージ再設計（ToBe）

結論を決める（どちらか）。

### 8.1 方針A：Google Drive継続（ただし運用品質まで）
- 方式
  - Shared Drive + Service Account（推奨：組織運用が可能な場合）
  - もしくは User OAuth（refresh_token）をテナント別に保持
- 必要機能
  - フォルダ構成（tenant/year_month/employee など）
  - 権限（公開リンク禁止を基本、管理UIからの閲覧方法を設計）
  - 失敗時のリトライ・再アップロード導線

### 8.2 方針B：S3互換（R2/S3/GCS）へ移行（推奨：SaaSとして堅牢）
- 方式
  - 画像はオブジェクトストレージに保存し、管理UIは**署名付きURL**で閲覧
  - 画像の正規化（縮小/圧縮）、マルウェアスキャン（可能なら）
- メリット
  - テナント分離、容量/権限/監査がやりやすい
  - DriveのUIや権限に依存しない

## 11. マイルストーン（1か月版の現実プラン）

### Week 1：土台の作り直し（最重要）
- テナント/設定/管理ユーザー/RBAC のDB設計・実装
- `/admin/integrations` の骨格（設定の保存と表示）
- 監査ログの項目整理（イベント追加）

### Week 2：連携の本実装
- freee OAuth（HTTPS callback）実装 + トークン暗号化保存
- ストレージ方針確定（Drive継続 or S3互換）し、ToBeの実装へ寄せる

### Week 3：月次処理の運用品質
- 冪等・再実行・失敗理由の保持/表示
- 実行履歴と通知（最低限：管理画面で失敗が見える）
- 可能ならCron（「未実行通知」→余力で自動実行）

### Week 4：品質仕上げ
- バリデーション強化（画像/金額/日付/文字数）
- E2E動作確認シナリオ整備（職人→管理→freee）
- 監査/セキュリティ（Cookie/権限/ログ）最終点検

## 13. 未決事項（早めに決める）
- ストレージは Drive 継続か、S3互換へ移行か
- レビューは **月次（テナント×月）でまとめて承認**する方針にする  
  - 申請1件ずつの「承認/差戻し」UIは作らない（事務局が迷う/運用コスト増のため）  
  - 代わりに、申請には `needs_review` 等の**フラグ付け（自動/AI）**だけを持たせ、月次画面で「要確認のみ」を見せる
- テナントのオンボーディングは **管理者（運営）が作成**（セルフサーブは後回し）
